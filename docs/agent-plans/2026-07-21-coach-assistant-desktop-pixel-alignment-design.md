# 教练助手桌面端像素级视觉对齐设计

## 1. 背景

项目已完成欢迎页、员工输入、类型判定、方案生成和辅导反馈的第一轮视觉改造，并保留真实 API、四画像改选、返回导航、GROW/SBI、安全 Markdown 渲染和当前会话状态。

甲方要求下一轮以以下参考源码作为桌面端视觉真源：

`docs/管理自我和管理团队两个智能体/教练助手 .html`

本设计采用“受控重建视图层”：逐页复刻参考源码的桌面 DOM 结构和 CSS，但继续使用当前应用的数据、状态与事件处理。不得直接移植参考源码中的静态数据、模拟 AI、内联 `onclick` 或 `innerHTML` 渲染。

## 2. 目标与验收基准

### 2.1 目标

- 欢迎页和四个流程阶段在桌面端完整对齐甲方参考 HTML。
- 视觉对齐不能破坏现有真实功能或安全边界。
- 对齐结果能够通过同环境截图和关键几何数据复核，而不是仅凭主观目测。

### 2.2 固定验收环境

- Chromium 浏览器。
- 浏览器缩放 100%。
- 页面视口 `1920 × 1080`。
- 参考 HTML 与应用使用同一浏览器、同一视口分别截图。
- 关键元素的几何差异目标不超过约 `2px`；字体栅格化造成的细微差异不作为结构缺陷。

### 2.3 桌面端边界

- 本轮只修改桌面端。
- 新增或调整的桌面规则放入 `@media (min-width: 821px)` 或等价桌面选择器。
- 现有移动端结构和样式不调整；只要求本轮修改不主动破坏当前移动端行为。

## 3. 已确认需求

### 3.1 要求 1：欢迎页品牌区与介绍文案

品牌区以参考 HTML 为准：

- 橙色品牌图标使用白色人物线框图标，不再显示“教”字。
- “教练助手”和“能力 × 意愿画像 · 差异化辅导”上下两行排列。
- 对齐图标尺寸、品牌字号、副标题字号、行高、左右间距和顶部留白。

首页介绍文案使用完整内容：

> 描述一位待辅导员工，AI 按“能力 × 意愿”匹配 4 类画像，输出差异化的沟通与教练方案：说什么、注意什么、多久沟通一次、如何修正绩效差距。

文案容器宽度、字号、行高、上下间距和两行换行效果以参考页面在固定验收环境中的渲染结果为准。

### 3.2 要求 2：欢迎页四步流程颜色

- 四个流程标签使用参考 HTML 的淡橙色背景。
- 标签文字使用橙棕色。
- 箭头使用参考 HTML 的浅灰紫色。
- 流程内容仍为“信息输入 → 类型判定 → 方案生成 → 辅导反馈”。
- 本项只调整视觉，不改变入口行为。

### 3.3 要求 3：员工信息输入页

工作区页头：

- 增加参考 HTML 的返回箭头按钮。
- 显示“教练助手”标题和“管理团队”标签。
- 返回箭头复用现有返回首页/会话重置路径，取消未完成请求并清空当前会话。

表单结构分为三个编号区域：

1. 员工基础信息：岗位类别、在团队入职时长、当前绩效状态。
2. 目标与困扰：绩效目标/上层期望、近期辅导困扰。
3. 员工特征描述：关键词多选和自由补充文本。

固定关键词为：

- 学习能力强
- 执行力弱
- 主动性不足
- 情绪易波动
- 沟通抵触
- 责任心强
- 经验不足
- 追求稳定
- 有上进心
- 需要认可

关键词行为：

- 支持鼠标和键盘多选、取消选择。
- 选中状态不能只依靠颜色表达，应有边框、背景或 `aria-pressed` 状态。
- 所选关键词与自由补充文本在提交时合并为现有 `intake.traits` 字符串。
- 不向 `/api/coach/intake` 增加字段，不修改服务端契约。
- 返回员工输入页时恢复关键词和自由文本。
- 返回首页、完成辅导或刷新页面后清空。

底部操作栏对齐参考 HTML，主按钮文案为“判定类型 →”。信息不足时仍按现有流程展示 AI 追问，不得绕过步骤 1 的充分性审查。

### 3.4 要求 4：四个流程页面完整对齐

#### 类型判定

- 对齐左侧完成态、当前态和未完成态步骤样式。
- 对齐“节点 ② · AI动作”、标题和说明文字。
- 显示参考提示条、四画像卡片、推荐标记、判定依据和底部操作栏。
- 四画像名称保持当前已确认名称：熟手待激活型、核心明星型、潜力新兵型、待改进型。
- 用户仍可改选；后续方案严格使用最终选择。
- 前台继续隐藏 `D1/D2`，内部映射逻辑不变。
- 待补充或待人工确认不能通过卡片选择进入方案生成。

#### 教练方案生成

- 对齐“节点 ③ · 输出”、标题、动态画像名称说明和主面板布局。
- 方案内容按参考 HTML 展示为五类卡片：沟通切入点、沟通注意事项、建议沟通频率、绩效差距修正方法、话术示例。
- 每张卡显示参考 HTML 的单字图标、标题层级、边框、圆角、内边距和列表圆点。
- 底部操作栏对齐“上一步、复制方案、换个角度、去反馈 →”。
- GROW/SBI 内容和阶段独立换行继续保留；视觉卡片不能删除或改写模型内容。

#### 辅导反馈

- 对齐“节点 ④ · 会话内迭代”、标题、说明、反馈输入框和生成按钮。
- 显示“会话内辅导记录”，内容来自当前内存中的最终画像和首次方案。
- 不新增数据库、浏览器持久化或跨会话历史。
- 反馈结果仍通过安全 Markdown 渲染器展示。
- 底部操作栏对齐“上一步、完成辅导 →”。
- “完成辅导”复用现有会话重置路径。

## 4. 架构设计

### 4.1 视觉模板与功能真源分离

- 参考 HTML 是视觉模板：提供 DOM 分组、文案层级、尺寸、色彩和间距。
- 当前应用是功能真源：提供 API、状态、错误处理、请求取消、返回导航和安全渲染。
- 不复制参考 HTML 的状态对象、模拟延时、静态输出或字符串模板渲染。

### 4.2 视图边界

`frontend/views.js` 继续负责安全创建 DOM：

- 原子字段使用 `textContent`。
- 模型自由文本继续交给 `renderMarkdown`。
- 事件继续使用 `addEventListener`。
- 不直接使用 `innerHTML` 渲染模型或用户内容。

可增加小型视图辅助函数，例如：

- 工作区页头。
- 编号分组标题。
- 特征关键词组。
- 方案卡片标题图标。
- 会话内记录项。

辅助函数只减少重复，不进行无关重构。

### 4.3 状态设计

在现有内存会话中增加两个前端字段：

- `selectedTraits: string[]`：已选关键词。
- `traitNote: string`：自由补充文本。

字段行为：

- 初始值分别为 `[]` 和空字符串。
- 选择关键词或输入补充文本时更新当前内存。
- 提交前通过纯函数合成 `intake.traits`。
- `clearDownstream('intake')` 只清理依赖结果，不错误清除刚提交的当前输入。
- `resetSession()`、返回首页、完成辅导和刷新初始化清空字段。
- 返回上一步不清空字段。

合并规则：

- 只有关键词：以中文顿号连接。
- 只有补充文本：直接使用补充文本。
- 两者都有：`关键词：A、B。补充描述：文本`。
- 两者都没有：传空字符串，继续由现有充分性审查决定是否追问。

### 4.4 会话内辅导记录

反馈页记录由已有状态派生，不新增持久状态：

- 类型判定记录：最终前台画像名称与能力/意愿。
- 首次方案记录：从当前 `plan` 的切入点、频率或关键建议生成简短可见摘要。

记录仅用于当前页面展示，不改变反馈 API 的 `planSummary` 逻辑。

## 5. 样式策略

### 5.1 直接复用参考 token

从参考 HTML 提取并对齐：

- 紫色、橙色、背景、表面、正文、弱化文字和边框色。
- 圆角、阴影、按钮高度和字体权重。
- 页面最大宽度、左右栏宽度与栏间距。
- 面板头部、正文、底部操作栏的内边距。

禁止使用大量绝对定位或按单张截图硬编码坐标。结构应依靠 grid、flex、padding 和 gap 达到同样布局。

### 5.2 桌面隔离

- 桌面端对齐规则集中在明确的桌面媒体查询中。
- 不删除现有移动端媒体查询。
- 若基础 token 必须调整，应确认不会导致移动端明显退化；否则用桌面局部变量覆盖。

## 6. 数据流

### 员工输入

1. 用户选择关键词并填写补充文本。
2. 前端保存 `selectedTraits` 与 `traitNote`。
3. 点击“判定类型”。
4. 前端合并为现有 `traits` 字符串。
5. 调用现有 intake 流程。
6. 信息不足时展示追问；充分时进入类型判定。

### 类型、方案与反馈

1. AI 返回原始有效分类。
2. 前端显示四画像并初始化最终选择。
3. 用户可无 API 改选。
4. 步骤 3 和步骤 4使用现有最终分类解析逻辑。
5. 方案和反馈继续走现有 API 与安全渲染。

## 7. 错误与安全边界

- 请求失败继续使用现有错误区域和重试路径。
- 进行中返回首页、返回上一步或完成辅导时取消请求，并通过 request epoch 阻止迟到响应恢复旧页面。
- 高风险人事事项继续显示固定 HR 提示。
- 不读取或展示 `.env`、API Key 或其他密钥。
- 自动化和截图验证全部使用 fixture/fake API，不产生真实模型费用。
- 不使用 `localStorage`、`sessionStorage` 或数据库。

## 8. 验证设计

### 8.1 TDD

按页面顺序执行：

1. 欢迎页与品牌区。
2. 员工输入页与关键词状态。
3. 类型判定页。
4. 方案生成页。
5. 辅导反馈页。
6. 全量回归。

每页先增加失败测试，确认失败原因，再实现最小变更。

### 8.2 视觉验证

每页在固定环境下生成两张截图：

- 参考 HTML 页面。
- 当前应用对应页面。

检查：

- 页面与主容器位置。
- 左侧步骤栏和右侧面板尺寸。
- 字号、行高、颜色、圆角、边框与阴影。
- 内容卡片和底部操作栏。
- 文案换行与按钮排列。

使用 `getBoundingClientRect()` 和 `getComputedStyle()` 对关键元素做可量化断言，目标误差约 `2px`。不引入新的图像比较依赖。

### 8.3 功能回归

必须继续通过：

- 员工资料充分性追问。
- 四画像选择与最终画像请求负载。
- D1/D2 隐藏映射。
- 返回上一步、返回首页和请求取消。
- 反馈文本保留及刷新/回首页清空。
- GROW/SBI 阶段展示。
- Markdown HTML 转义与危险链接拦截。
- 高风险 HR 拦截。
- 桌面端无整页横向溢出。

## 9. 非目标

- 不修改移动端视觉。
- 不修改服务端模型契约、Prompt、知识库或 DeepSeek 请求逻辑。
- 不把 D1/D2 合并为服务端 D。
- 不新增依赖、框架、数据库、登录、员工档案或跨会话历史。
- 不重新设计甲方已确认的视觉风格。

## 10. 完成标准

- 欢迎页和四个流程阶段在固定桌面环境下与参考 HTML 达到结构和视觉对齐。
- 要求 1—4全部落实。
- 关键词选择真实参与 intake 请求，并保持现有 API 格式。
- 四画像、最终选择、方案、反馈和会话记录逻辑一致。
- 全量自动化测试通过。
- 提供参考页与应用页的最终同尺寸截图供甲方人工验收。
