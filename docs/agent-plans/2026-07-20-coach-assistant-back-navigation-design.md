# 教练助手“返回上一步”功能设计

> 状态：设计已确认，待生成实施计划。
> 范围：辅导流程第 2、3、4 步的前台导航与会话内状态一致性。
> 不涉及：后端接口、数据库、跨会话记忆、HR/IM 集成和模型提示词调整。

## 1. 目标

在员工信息输入、类型判定、教练方案生成、辅导反馈四步流程中，为第 2、3、4 步提供统一的“返回上一步”按钮，使管理者可以检查或修正前序信息，同时避免新输入与旧判定、旧方案或旧反馈混用。

## 2. 交互设计

- 第 2 步“类型判定”点击“返回上一步”，进入第 1 步“员工信息输入”。
- 第 3 步“教练方案生成”点击“返回上一步”，进入第 2 步“类型判定”。
- 第 4 步“辅导反馈”点击“返回上一步”，进入第 3 步“教练方案生成”。
- 按钮位于各步骤底部操作区左侧，使用次要按钮样式；当前步骤的主要操作按钮保持在右侧。
- 第 1 步不显示“返回上一步”。
- 返回操作本身不调用后端 API，也不清除已填写内容或已生成结果。
- 移动端允许底部按钮自然换行，不产生页面横向滚动。

## 3. 会话状态规则

### 3.1 仅返回查看

如果用户返回后没有修改或重新提交任何前序数据，系统保留当前会话中的员工信息、类型判定、教练方案、反馈输入和反馈结果。再次前进时可继续显示已有结果。

### 3.2 修改后重新提交

系统为每个已提交阶段保存仅存在于浏览器内存中的输入快照。重新提交时比较新旧快照；只有实际内容发生变化时，才按依赖关系清除后续结果：

- 员工信息或追问答案变化：清除信息审查结果、类型判定、教练方案、反馈输入和反馈结果，再执行新的信息审查。
- 类型判定所依据的结构化画像变化：清除旧类型判定、教练方案、反馈输入和反馈结果，再执行新的类型判定。
- 重新生成或“换个角度”生成教练方案：保留员工信息与类型判定，清除旧反馈输入和反馈结果。
- 仅返回查看且输入快照未变化：不清除任何结果，也不重复调用 API。

所有快照均只保存在当前 JavaScript 运行内存中；刷新、返回首页或重新开始后与其他会话状态一起清除。

## 4. 请求与导航安全

- 点击“返回上一步”时，取消当前页面尚未完成的请求，并将 `busy` 状态恢复为 `false`。
- 递增当前请求 epoch，使返回前发出的迟到响应无法覆盖返回后的页面或会话数据。
- 返回按钮在请求进行中仍可使用，作为用户主动离开当前步骤的方式。
- 返回后发生 API 错误时，仅显示在当前步骤，不恢复已经失效的后续结果。

## 5. 代码边界

- `frontend/views.js`：在第 2、3、4 步底部操作区渲染统一的“返回上一步”按钮。
- `frontend/app.js`：提供统一的 `goPrevious` 导航处理器，负责取消请求、切换步骤和触发重新渲染。
- `frontend/state.js`：保存阶段输入快照，并提供按依赖关系清理后续状态的函数。
- `frontend/api.js`：复用现有取消请求和 request epoch 机制，不新增接口。
- `tests/frontend.spec.js`：覆盖导航、保留、失效清理、迟到响应和窄屏行为。

不新增后端路由；现有 `/api/coach/intake`、`/classify`、`/plan`、`/feedback` 请求格式保持不变。

## 6. 错误与边界场景

- 第 2 步处于“待补充”或“待人工确认”状态时，仍可返回第 1 步补充信息。
- 第 4 步已生成反馈后返回第 3 步，旧反馈继续保留；只有方案发生重新生成时才清除。
- 第 3 步返回第 2 步后，如果类型判定没有变化，旧方案继续保留。
- 连续快速点击返回按钮不会让步骤小于 1，也不会触发重复 API 调用。
- 过期请求返回成功或失败时均不得改变当前页面。

## 7. 验收标准

- [ ] 第 2、3、4 步均显示“返回上一步”，第 1 步不显示。
- [ ] 按钮分别返回相邻前一步，不跳过步骤、不返回首页。
- [ ] 单纯返回查看不会调用 API，也不会清除已有输入和结果。
- [ ] 员工信息变化并重新提交后，旧类型、方案和反馈被清除。
- [ ] 类型判定输入变化并重新提交后，旧方案和反馈被清除。
- [ ] 重新生成方案后，旧反馈输入和反馈结果被清除。
- [ ] 返回操作会取消未完成请求，迟到响应不会恢复旧页面或旧数据。
- [ ] 刷新与返回首页仍按现有规则清空全部会话状态。
- [ ] 320px 和 375px 窄屏下按钮可见且页面无横向溢出。
- [ ] 不新增后端接口、浏览器持久化或跨会话数据。
